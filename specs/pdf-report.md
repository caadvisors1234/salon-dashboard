# Spec: PDFレポート出力

**ステータス**: COMPLETED
**作成日**: 2026-02-16
**最終更新**: 2026-02-16
**要件定義書参照**: セクション 6（レポート出力仕様）、セクション 5.1（技術スタック）、セクション 7（非機能要件）、セクション 8 Phase 4

---

## 1. 目的

代理店・コンサルタントがクライアントへ提出するパフォーマンスレポートをPDFとして生成・ダウンロードできる機能を構築する。クライアント自身も自社レポートをダウンロード可能とする。

要件定義書セクション6より:
- 出力単位: **店舗単位** / **クライアント単位**
- 期間指定: 任意の月範囲
- 含有内容: GBP KPI + グラフ + キーワードテーブル + HPBセクション（データあり時）
- ヘッダー: 代理店ロゴ + クライアント名
- 生成方法: ダウンロードボタン押下で生成
- 同時生成制限: キュー処理 + 最大2件（初期設定）

## 2. 要件

### 機能要件

#### レポート生成UI（画面 No.5）

- [ ] FR1: 店舗詳細ダッシュボード（No.4）に「レポート出力」ボタンを配置する
- [ ] FR2: クライアント詳細（No.3）に「一括レポート出力」ボタンを配置する
- [ ] FR3: レポート出力ダイアログで以下を設定可能とする:
  - 出力単位: 店舗単位 / クライアント単位（クライアント詳細からの場合のみ選択可能）
  - 期間: 開始年月 〜 終了年月（MonthPicker）
- [ ] FR4: 「生成」ボタン押下でPDF生成をリクエストし、生成中はプログレス表示する
- [ ] FR5: 生成完了後、自動ダウンロードを開始する

#### PDF内容

- [ ] FR6: **ヘッダー**: 代理店ロゴ（静的ファイル `public/logo.png`）+ クライアント名 + 対象期間
- [ ] FR7: **店舗単位レポート**: 以下のセクションを含む
  - 店舗名・基本情報
  - GBP KPIサマリー（総合評価、レビュー数、合計閲覧数、合計アクション数 + 前月比）
  - 閲覧数推移グラフ（4系列折れ線・指定期間）
  - アクション指標推移グラフ（3系列折れ線・指定期間）
  - デバイス内訳円グラフ（最終月）
  - 検索キーワードランキングテーブル（最終月・上位20件）
  - HPBセクション: KPIカード + 推移グラフ（データあり時のみ）
- [ ] FR8: **クライアント単位レポート**: 店舗ごとの個別PDFを生成し、ZIPファイルとしてまとめてダウンロード
- [ ] FR9: HPBデータ未登録店舗はHPBセクションを省略する
- [ ] FR10: ページサイズはA4横（ダッシュボードのグラフレイアウトに適合）

#### 同時生成制限

- [ ] FR11: PDF生成リクエストをキューで管理し、同時実行数を最大2件に制限する
- [ ] FR12: キュー待ち状態のジョブは「待機中（N番目）」と表示する
- [ ] FR13: 生成中のジョブはタイムアウト（5分）を設定し、超過時はエラーとする

#### ファイル管理

- [ ] FR14: 生成したPDFファイル名: `{クライアント名}_{店舗名}_{期間}.pdf`（店舗単位）
- [ ] FR15: クライアント単位ZIPファイル名: `{クライアント名}_レポート_{期間}.zip`（中に店舗別PDFを格納）

### 非機能要件

- [ ] NFR1: パフォーマンス - 店舗単位レポート生成: 30秒以内
- [ ] NFR2: パフォーマンス - クライアント単位レポート（10店舗）: 3分以内
- [ ] NFR3: メモリ - Puppeteerインスタンスのメモリ使用量を監視し、1GB超過時は生成を拒否
- [ ] NFR4: セキュリティ - RLSポリシーに基づき、アクセス権のあるデータのみレポート化可能
- [ ] NFR5: 信頼性 - 生成失敗時はエラーメッセージを返却し、キューから除去する

### スコープ外

- レポートの定期自動生成（スケジュール配信）
- レポートのメール送信
- 生成済みPDFのSupabase Storageへの永続保存（オンデマンド生成のみ）
- カスタムレポートテンプレート（テンプレート選択UI）
- レポート生成履歴の管理画面

### 権限マトリクス

| 操作 | Admin | Staff | Client |
|------|-------|-------|--------|
| 店舗単位レポート生成 | ✅（全店舗） | ✅（担当店舗のみ） | ✅（自社店舗のみ） |
| クライアント単位レポート生成 | ✅（全クライアント） | ✅（担当クライアントのみ） | ✅（自社のみ） |

## 3. 技術設計

### 使用技術

- **PDF生成**: Puppeteer（Docker環境でChromiumを利用）
- **ZIP生成**: archiver（クライアント単位レポートの店舗別PDFまとめ）
- **レポートテンプレート**: 専用Next.jsページ（`/report/store/[locationId]`）をPuppeteerでレンダリング → PDF化
- **キュー**: インメモリキュー（`src/lib/pdf/queue.ts`）+ 同時実行制限（Semaphore パターン）
- **チャート**: Recharts（既存コンポーネントを再利用）
- **UI**: shadcn/ui Dialog + Progress

### アーキテクチャ概要

```
[ブラウザ] → [API: POST /api/reports/generate]
                ↓
           [キュー投入]
                ↓
           [Semaphore（最大2並列）]
                ↓
           [Puppeteer起動]
                ↓
           [レポートページ取得: /report/store/{locationId}?from=...&to=...]
                ↓
           [PDF生成 → バッファ返却]
                ↓
           [レスポンス: PDFバイナリ]
```

### APIエンドポイント

| メソッド | パス | 権限 | 説明 |
|---------|------|------|------|
| POST | `/api/reports/generate` | Admin/Staff/Client | PDF生成リクエスト。レスポンスはPDFバイナリ（`Content-Type: application/pdf`） |
| GET | `/api/reports/queue-status` | Admin/Staff/Client | 現在のキュー状態（待機数、実行中数） |

#### POST `/api/reports/generate`

**リクエストボディ:**
```typescript
{
  type: 'store' | 'client';       // 出力単位
  locationId?: string;             // 店舗単位の場合
  orgId?: string;                  // クライアント単位の場合
  startMonth: string;              // 'YYYY-MM'
  endMonth: string;                // 'YYYY-MM'
}
```

**レスポンス:**
- 成功（店舗単位）: `200 OK` + `Content-Type: application/pdf` + PDFバイナリ
- 成功（クライアント単位）: `200 OK` + `Content-Type: application/zip` + ZIPバイナリ
- キュー満杯: `429 Too Many Requests` + `{ error: "生成キューが満杯です", queuePosition: N }`
- エラー: `500 Internal Server Error` + `{ error: "PDF生成に失敗しました" }`

### レポートテンプレートページ

内部専用ページ（外部アクセス不可）として以下を作成:

| パス | 用途 |
|------|------|
| `/report/store/[locationId]` | 店舗単位レポートテンプレート |

- クエリパラメータ: `from`, `to`（期間）、`token`（内部認証トークン）
- Puppeteerからのみアクセスされる想定
- 印刷用CSSでA4横レイアウトに最適化
- グラフはRechartsで描画（Puppeteerがレンダリング待機）

### レポートページの内部認証

Puppeteerからレポートページにアクセスする際、通常のSupabase Auth セッションは使えない。代わに:

1. API側で短命の内部トークン（JWT、有効期限: 5分）を生成
2. クエリパラメータ `?token=xxx` でレポートページに渡す
3. レポートページ側でトークンを検証し、含まれるユーザーIDでSupabase adminクライアント経由でデータ取得

### 主要コンポーネント・モジュール

#### バックエンド（`src/lib/pdf/`）

1. **`queue.ts`** - PDF生成キュー管理
   - `PdfQueue` クラス: キュー投入、Semaphoreによる同時実行制限（最大2）
   - `enqueue(job)` → `Promise<Buffer>`: キュー投入〜生成完了まで待機
   - `getStatus()` → `{ running: number, waiting: number }`

2. **`generator.ts`** - Puppeteer PDF生成ロジック
   - `generateStorePdf(locationId, startMonth, endMonth, token)` → `Buffer`
   - `generateClientZip(orgId, startMonth, endMonth, token)` → `Buffer`（店舗別PDF → ZIP）
   - Puppeteer起動 → ページ遷移 → `networkidle0` 待機 → `page.pdf()` → バッファ返却

3. **`token.ts`** - 内部認証トークン生成・検証
   - `createReportToken(userId, scope)` → `string`
   - `verifyReportToken(token)` → `{ userId, scope }`

#### APIルート（`src/app/api/reports/`）

4. **`generate/route.ts`** - PDF生成APIエンドポイント
5. **`queue-status/route.ts`** - キュー状態取得

#### レポートテンプレート（`src/app/report/`）

6. **`store/[locationId]/page.tsx`** - 店舗レポートテンプレートページ
7. **`layout.tsx`** - レポート用レイアウト（印刷用CSS、ヘッダー/フッター）
8. **`components/report-header.tsx`** - ヘッダー（ロゴ + クライアント名 + 期間）
9. **`components/report-gbp-section.tsx`** - GBPセクション（KPI + グラフ + キーワード）
10. **`components/report-hpb-section.tsx`** - HPBセクション

#### フロントエンドUI

11. **`src/components/dashboard/report-dialog.tsx`** - レポート出力ダイアログ（期間選択 + 生成ボタン + プログレス）

### データフロー

**店舗単位:**
```
1. ユーザーが店舗詳細の「レポート出力」ボタンをクリック
2. ダイアログで期間を選択し「生成」
3. POST /api/reports/generate { type: 'store', locationId, startMonth, endMonth }
4. API側:
   a. 認証チェック（セッション + RLS権限）
   b. 内部トークン生成
   c. キュー投入（Semaphore待機）
   d. Puppeteer起動
   e. レポートページ（localhost:3000/report/store/xxx?from=...&to=...&token=...）を取得
   f. page.pdf() でPDF生成
   g. Puppeteer終了
   h. PDFバッファをレスポンスとして返却
5. フロントエンド: Blobとしてダウンロード（.pdf）
```

**クライアント単位:**
```
1. ユーザーがクライアント詳細の「一括レポート出力」ボタンをクリック
2. ダイアログで期間を選択し「生成」
3. POST /api/reports/generate { type: 'client', orgId, startMonth, endMonth }
4. API側:
   a. 認証チェック
   b. 対象クライアントの全店舗一覧を取得
   c. 店舗ごとにPDF生成（キュー内で順次処理）
   d. 全PDFをZIPにまとめる（archiver）
   e. ZIPバッファをレスポンスとして返却
5. フロントエンド: Blobとしてダウンロード（.zip）
```

### 設計判断

| 判断事項 | 選択 | 理由 |
|---------|------|------|
| PDF生成方法 | Puppeteerでレポートページをレンダリング | 要件定義書指定。既存ダッシュボードのUI/グラフをそのまま活用可能 |
| レポートテンプレート | 専用Next.jsページ | ダッシュボードコンポーネントを再利用しつつ、印刷向けレイアウトに最適化 |
| キュー方式 | インメモリ Semaphore | 初期100店舗規模では十分。永続化が必要になればDB方式へ移行 |
| 同時実行上限 | 2件（設定可能） | 要件定義書指定。環境変数 `PDF_MAX_CONCURRENT` で変更可能に |
| PDF永続化 | しない（オンデマンド生成） | 要件が明示されていない。ストレージコスト回避。必要時に再生成 |
| ページサイズ | A4横 | グラフ・テーブルの視認性確保 |
| クライアント単位レポート | 店舗別PDFをZIPでダウンロード | 個別ファイルの再配布・差し替えが容易 |
| レポートページ認証 | 短命内部JWT | Puppeteer経由でのアクセスにSupabase Authセッションは使えないため |
| Puppeteer実行環境 | appコンテナ内（初期） | 要件定義書: 「負荷が高い場合は専用ワーカー分離」。初期はシンプルに |

## 4. 受入条件

### レポート生成

- [ ] AC1: 店舗詳細画面の「レポート出力」ボタンからPDFを生成・ダウンロードできる
- [ ] AC2: クライアント詳細画面の「一括レポート出力」ボタンから全店舗PDFをZIPでダウンロードできる
- [ ] AC3: 期間を指定（開始年月〜終了年月）でき、指定期間のデータがレポートに反映される
- [ ] AC4: ヘッダーに代理店ロゴ・クライアント名・対象期間が表示される
- [ ] AC5: GBP KPIカード（4種）が前月比付きで正しく表示される
- [ ] AC6: 閲覧数・アクション推移グラフが指定期間で正しく描画される
- [ ] AC7: デバイス内訳円グラフが最終月のデータで描画される
- [ ] AC8: キーワードランキングが上位20件表示される（threshold対応含む）
- [ ] AC9: HPBデータが存在する店舗ではHPBセクションが表示される
- [ ] AC10: HPBデータが存在しない店舗ではHPBセクションが省略される

### 同時生成制限

- [ ] AC11: 3件目のPDF生成リクエストがキュー待ちになる（同時2件制限）
- [ ] AC12: キュー待ち中に待機順が表示される
- [ ] AC13: 5分を超える生成はタイムアウトエラーになる

### 権限

- [ ] AC14: Admin は全店舗・全クライアントのレポートを生成できる
- [ ] AC15: Staff は担当クライアントの店舗のみレポート生成できる
- [ ] AC16: Client は自社店舗のレポートのみ生成できる
- [ ] AC17: 権限外の店舗・クライアントへのレポート生成リクエストは403エラーとなる

### 品質

- [ ] AC18: 店舗単位レポートが30秒以内に生成される
- [ ] AC19: PDFがA4横サイズで出力される
- [ ] AC20: グラフが解像度問題なくPDFに出力される（デバイススケール2x）
